# Umka Player — Manual Verification Guide (Standard v1.26.5.1)

This is a copy-paste-ready manual QA guide for verifying the reference Player against
**Umka Kiosk Standard v1.26.5.1**. It uses the `mosquitto_pub` / `mosquitto_sub` CLI tools
to send commands and observe what the Player publishes on MQTT. No coding required.

It covers happy paths (Browse round-trip, Loop, trigger pipeline, each command), edge cases
(wrong-shape payloads ignored, one-shot `triggerEnded`), and the two-tier liveness and error
states.

## Before You Start

You need two placeholders, set once and reused in every command below:

- `{broker}` — the MQTT broker host (and port if not 1883), e.g. `broker.example.local:1883`.
  This is the broker the Player is configured to talk to (its `mqttUrl` in `settings.json`,
  minus the `mqtt://` prefix).
- `{slug}` — the kiosk's slug (its `kioskSlug` in `settings.json`), e.g. `kiosk-main-hall`.

Export them in your shell so you can copy-paste the rest verbatim:

```bash
export BROKER_HOST=broker.example.local  # change to your broker host
export BROKER_PORT=41883               # change to your broker port (1883 is the MQTT default)
export SLUG=kiosk-main-hall            # change to your kiosk slug
```

Install the tools if needed: `sudo apt install mosquitto-clients` (Debian/Ubuntu).

**Base topic convention.** Every topic for a kiosk hangs off
`umka/kiosks/{slug}/...`. Commands are *sent* to `.../commands/{leaf}`; the Player *publishes*
its retained state to `.../status`, its app heartbeat to `.../heartbeat`, and (because this is
the all-in-one build that emulates the Supervisor) a system heartbeat to `.../system/heartbeat`.

If the broker requires auth, add `-u <user> -P <pass>` to every `mosquitto_pub`/`mosquitto_sub`.

## Setup: Two Observer Windows

Open **two terminals** and leave them running for the whole session.

**Terminal A — watch status (retained state):**

```bash
mosquitto_sub -h "$BROKER_HOST" -p "$BROKER_PORT" -v -t "umka/kiosks/$SLUG/status"
```

**Terminal B — watch liveness (app + system heartbeats):**

```bash
mosquitto_sub -h "$BROKER_HOST" -p "$BROKER_PORT" -v \
  -t "umka/kiosks/$SLUG/heartbeat" \
  -t "umka/kiosks/$SLUG/system/heartbeat"
```

The `-v` flag prints `topic payload` so you can see which topic each message arrives on. Because
`status` and `system/heartbeat` are **retained**, a freshly opened subscriber immediately
receives the last-published message — useful for reading current state at any time.

Send all commands from a **third** terminal (the export block above must be run there too).

Tip: to read the *current* retained status once without leaving a subscriber open, run the
Terminal A command and press Ctrl-C after the first line prints.

## Status Payload Cheat-Sheet

A `status` payload always contains: `kioskId`, `state`, `mode`, `volume`, `locale`,
`timestamp`, `version`, `uptime`, `error` (object or `null`). It *optionally* contains:
`currentContent`, `navigation` (Browse mode only: `{ nodeId, path[], showcaseOpen }`),
`screensaverActive`, and `triggerEnded` (one-shot `true` after a triggered play ends).

To pretty-print a status line, pipe it through `jq`:

```bash
mosquitto_sub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/status" -C 1 | jq
```

## Browse Round-Trip (Navigation)

Verifies `navigation.nodeId`, `navigation.path[]`, and `navigation.showcaseOpen` track the
visitor as they move through the catalog. The kiosk must be in Browse mode for this scenario
(`status.mode` should read `browse`; if not, see *Mode Switch* below).

Use real IDs from your content package where the steps say "a section" / "an object".

1. **Wake** (touch the screensaver). On a kiosk this is a physical tap; via MQTT, send a
   playback command that wakes it, then read status:
   ```bash
   mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
     -t "umka/kiosks/$SLUG/commands/playback" -m '{"action":"play"}'
   ```
   - Watch Terminal A. Expect `state` to leave `idle`/screensaver and `screensaverActive:false`.
     `navigation.nodeId` should be `null` at the catalog root, `navigation.path` `[]`.

2. **Enter a section** (open a submenu) — tap the section on the kiosk, or drive it with a
   `content` command pointing at the submenu's menu-item ID:
   ```bash
   mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
     -t "umka/kiosks/$SLUG/commands/playback" \
     -m '{"action":"content","contentId":"<section-menu-item-id>"}'
   ```
   - Expect `navigation.nodeId` = the section's ID, and `navigation.path` to include it
     (e.g. `["<section-id>"]`).

3. **Open an object** inside the section:
   ```bash
   mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
     -t "umka/kiosks/$SLUG/commands/playback" \
     -m '{"action":"content","contentId":"<object-menu-item-id>"}'
   ```
   - Expect `navigation.path` to grow to `["<section-id>","<object-id>"]` and
     `navigation.nodeId` = `<object-id>`. If the object is a showcase, expect
     `navigation.showcaseOpen:true`; otherwise `false`. `currentContent` should describe the object.

4. **Back** one level:
   ```bash
   mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
     -t "umka/kiosks/$SLUG/commands/playback" -m '{"action":"prev"}'
   ```
   (On the kiosk this is the Back button.) Expect `navigation.path` to shrink back to
   `["<section-id>"]`, `showcaseOpen:false`.

5. **Home** — return to the catalog root:
   ```bash
   mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
     -t "umka/kiosks/$SLUG/commands/playback" -m '{"action":"home"}'
   ```
   - Expect `navigation.nodeId:null`, `navigation.path:[]`, `showcaseOpen:false`.

**Edge check:** while in Browse, `navigation` must be present. After switching to Loop or
Custom mode, `navigation` should be **absent** from the status payload.

## Loop Playback

Verifies the playlist advances on media end and that `triggerEnded` is **not** present.

1. Put the kiosk in Loop mode (see *Mode Switch*) and start playback:
   ```bash
   mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
     -t "umka/kiosks/$SLUG/commands/playback" -m '{"action":"play"}'
   ```
2. Watch Terminal A: `state:"playing"`, `currentContent` shows playlist item 0.
3. Let the first media item play to the end (or send `{"action":"next"}` to simulate).
   - Expect `currentContent` to advance to the next playlist item, then loop to item 0
     after the last item.
   - **Assert:** `triggerEnded` is absent (never `true`) for ordinary loop advancement.

## Trigger Pipeline

Verifies a `trigger_play` plays the enveloped media and emits exactly one `triggerEnded:true`.

1. Send a `trigger_play` with the full media envelope (substitute a real cached media URL/ID):
   ```bash
   mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
     -t "umka/kiosks/$SLUG/commands/playback" \
     -m '{"action":"trigger_play","mediaId":"trigger-clip-1","mediaUrl":"media-cache://local/trigger-clip-1.mp4","mediaMimeType":"video/mp4","mediaTitle":"Triggered Clip"}'
   ```
   - Expect `state:"playing"`, `currentContent.id:"trigger-clip-1"`, `screensaverActive:false`.
2. Let the clip play to the end.
   - **Assert:** exactly **one** status message appears with `triggerEnded:true`, at the
     moment the clip finishes. State returns to `idle`/screensaver (or `menu` in Browse).
   - **Assert:** the *next* status message (any subsequent change) no longer carries
     `triggerEnded:true` — it is one-shot.

Count occurrences over a window to be sure:

```bash
mosquitto_sub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/status" -W 20 \
  | grep -c '"triggerEnded":true'
```

Send the `trigger_play` above while this runs; it should print `1`.

## Commands (Exact Payloads)

For each, send the command and confirm the matching field changes in Terminal A's status.

**Volume** — bare integer (topic `commands/volume`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/volume" -m '75'
```
Expect `volume:75`.

**Locale** — bare or JSON string (topic `commands/locale`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/locale" -m 'en'
# or quoted JSON form:
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/locale" -m '"en"'
```
Expect `locale:"en"`.

**Loop** — bare boolean (topic `commands/loop`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/loop" -m 'true'
```
Looping is internal state; confirm no error and that subsequent loop playback wraps.

**Screensaver** — force idle/screensaver (topic `commands/playback`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
  -t "umka/kiosks/$SLUG/commands/playback" -m '{"action":"screensaver"}'
```
Expect `screensaverActive:true` and `state:"idle"`.

**Seek** — JSON with `value` in seconds (topic `commands/playback`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
  -t "umka/kiosks/$SLUG/commands/playback" -m '{"action":"seek","value":30}'
```
With media playing, the video jumps to ~30s; status re-publishes (no error).

**Mode** — JSON on `commands/app` (topic `commands/app`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
  -t "umka/kiosks/$SLUG/commands/app" -m '{"action":"mode","value":"loop"}'
```
Expect `mode:"loop"` (try `"browse"` / `"custom"` too). Remember: `navigation` only appears in `browse`.

**Sync** — reload content from CMS (topic `commands/app`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
  -t "umka/kiosks/$SLUG/commands/app" -m '{"action":"sync"}'
```
Player re-fetches its content package; expect no error and content to reload.

**Restart** — JSON `{action:"restart"}` triggers a **renderer reload** (topic `commands/app`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
  -t "umka/kiosks/$SLUG/commands/app" -m '{"action":"restart"}'
```
Expect the app window to reload (uptime resets, fresh `loading` → ready). **Contrast:** the
Supervisor's bare-string `restart` is **ignored** by the Player — verify it does nothing:
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/app" -m 'restart'
```
Expect: nothing happens (logged and ignored — it is the Supervisor's, wrong shape for the Player).

**Quit** — graceful application quit (topic `commands/app`):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" \
  -t "umka/kiosks/$SLUG/commands/app" -m '{"action":"quit"}'
```
Expect the Electron app to exit, and a graceful-offline on `system/heartbeat` (see *Two-Tier
Liveness*). Run this **last**, since it closes the app.

## Power (Supervisor-Emulated)

`commands/power` is a Supervisor topic the all-in-one build emulates (`src/services/supervisor.ts`
plus Electron IPC). On a real production split this is owned by the Sentinel, not the Player.

**Reboot:**
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/power" -m 'reboot'
```
**Shutdown** (`off` and `shutdown` both mean power-off):
```bash
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/power" -m 'off'
```
Expect the OS reboot/shutdown command to run (Electron desktop build only). In browser-mode
dev there is no OS action; the command is accepted and logged. **These power off the machine —
run only when you intend to.**

## Two-Tier Liveness

Verifies the Supervisor-emulation heartbeat, the LWT on ungraceful death, and graceful-offline.

1. **System heartbeat cadence.** In Terminal B, watch `umka/kiosks/$SLUG/system/heartbeat`.
   - **Assert:** a retained message arrives roughly **every 10 seconds** with
     `player.status:"running"`, a `pid`, `restartCount:0`, `lastCrash:null`, and a `system`
     block (`cpuPercent`/`memoryPercent` are `0` in the reference build, `networkConnected`
     reflects MQTT connectivity). The app `heartbeat` topic also ticks every ~10s.

2. **Ungraceful death → LWT.** Kill the app process abruptly (no clean shutdown):
   ```bash
   pkill -9 -f umka            # adjust to match the Player process name
   ```
   - **Assert:** within the broker's keepalive window, a retained message lands on
     `system/heartbeat` with `status:"offline"` and **no** `graceful` field. This is the MQTT
     Last Will the broker fires on behalf of the dead client.

3. **Graceful quit → graceful-offline.** Start the app again, then quit it cleanly (via the
   `quit` command above, or closing the window):
   - **Assert:** a retained message lands on `system/heartbeat` with `status:"offline"` **and**
     `graceful:true`. The presence of `graceful:true` is what distinguishes a clean shutdown
     from a crash/yank.

   Read the current retained system-heartbeat at any time to inspect it:
   ```bash
   mosquitto_sub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/system/heartbeat" -C 1 | jq
   ```

Note: the reference build does **not** publish `system/crash` and does **not** emit Wake-on-LAN —
a running process cannot report its own crash or power itself on; those are the real Sentinel's
job in production.

## Error Path

Verifies the conformant error code surfaces on `status.error` with `state:"error"`.

The only error code the Player emits is `INDEXEDDB_OPEN_FAILED_PERMANENT` (after the IndexedDB
open-retry budget is exhausted). The other codes — `INDEXEDDB_OPEN_FAILED`, `CONTENT_SYNC_FAILED`,
`MEDIA_DOWNLOAD_FAILED`, `SETTINGS_LOAD_FAILED` — are reserved but not emitted.

1. Simulate an IndexedDB open failure. The simplest reproducible way in a dev/browser run:
   - Open the app in a browser context where IndexedDB is unavailable/blocked (e.g. a private
     window with storage disabled, or a browser policy that denies IndexedDB), **or**
   - In DevTools, override `indexedDB.open` to throw before the app initializes storage, then
     reload so the open path runs and exhausts its retries.

2. Read the retained status:
   ```bash
   mosquitto_sub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/status" -C 1 | jq
   ```
   - **Assert:** `state == "error"` and `error.code == "INDEXEDDB_OPEN_FAILED_PERMANENT"`,
     with a human-readable `error.message` and an ISO `error.timestamp`.
   - **Assert (healthy contrast):** in any normal status, `error` is `null`.

## Edge Cases: Wrong-Shape Payloads Are Ignored

The Player parses each command topic per its expected shape and **ignores** mismatches
(logged, no state change). Spot-check a few — none of these should change status:

```bash
# volume expects a bare integer — a JSON object is ignored
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/volume" -m '{"v":75}'
# loop expects bare true/false — "yes" is ignored
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/loop" -m 'yes'
# playback expects JSON {action} — a bare string is ignored
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/playback" -m 'play'
# app expects JSON {action} — bare strings (the Supervisor's) are ignored
mosquitto_pub -h "$BROKER_HOST" -p "$BROKER_PORT" -t "umka/kiosks/$SLUG/commands/app" -m 'stop'
```

**Assert:** Terminal A shows no change for any of these; the Player logs each as ignored.

## Quick Checklist

- [ ] Browse round-trip updates `navigation.nodeId` / `path` / `showcaseOpen` at each step
- [ ] `navigation` present only in Browse mode
- [ ] Loop advances on media end; `triggerEnded` absent
- [ ] `trigger_play` plays the envelope; exactly one `triggerEnded:true` on completion
- [ ] `volume` / `locale` / `loop` / `screensaver` / `seek` / `mode` / `sync` apply correctly
- [ ] `restart` (JSON) reloads renderer; bare-string `restart` ignored
- [ ] `quit` exits the app and publishes graceful-offline
- [ ] `power` `off` / `reboot` accepted (Supervisor-emulated)
- [ ] `system/heartbeat` every ~10s; LWT fires `status:"offline"` (no `graceful`) on kill;
      graceful-offline (`graceful:true`) on clean quit
- [ ] IndexedDB failure → `state:"error"`, `error.code == "INDEXEDDB_OPEN_FAILED_PERMANENT"`
- [ ] Wrong-shape payloads are ignored (no state change)
